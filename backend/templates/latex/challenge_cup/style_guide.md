# 挑战杯模版 Style Guide

本指南补充 LaTeX 模版，用于指导系统生成中文“科学假设与研究计划”。输出必须面向挑战杯/大学生科研竞赛语境：问题真实、逻辑清楚、技术路线可执行、数据来源合规、参考文献真实可核查。

## Reviewer Expectations

- 评审首先看“问题是否真实且值得研究”：必须指出当前领域的具体局限性，而不是泛泛描述趋势。
- 研究计划需要有清晰推导链条：从历史数据或已发表研究出发，提出可检验的科学假设，再落到可执行实验。
- 创新点应体现为新的问题切入、变量组合、技术路线、数据采集方式或验证框架，而不是只堆砌模型名称。
- 数据必须来源合规：优先使用公开数据集、政府/机构开放数据、合法授权数据或明确说明拟采集方案。
- 结果部分可以是初步实验、公式推导、可行性仿真或小样本验证，但必须说明验证范围和不确定性。
- 参考论文必须真实存在；不确定的文献不得写入最终 References。

## Common Content Structure

1. **待研究问题（Problem Statement）：** 指出领域背景、现有方法或现实应用中的明确局限，给出待解决问题边界。
2. **解决思路（Rationale）：** 用逻辑推理展示创新点，从历史事实、理论机制或数据观察推导到科学假设。
3. **必要的技术手段（Technical Details）：** 列出统计建模、机器学习、深度学习、因果推断、优化、实验平台或软件工具。
4. **数据集（Datasets）：** 写明真实数据集名称、来源、许可或合规性，以及数据规模、变量和适用性。
5. **Source：** 描述假设推演依据的历史数据、文献证据、公开统计或既有实验结果。
6. **Target：** 描述验证实验拟采集的数据特征，包括样本、变量、频率、标注、设备、伦理与隐私约束。
7. **标题（Paper Title）：** 使用规范中文学术标题，体现对象、方法和目标。
8. **摘要（Paper Abstract）：** 包含背景、问题、方法、预期结果和意义，避免竞赛口号式表达。
9. **方法论（Methods）：** 写出实施步骤、模型架构、实验流程、变量定义和质量控制。
10. **实验设计（Experiments）：** 包含 Baselines、Metrics、消融或对照组，说明如何验证假设。
11. **实验结果（Results）：** 通过公式推导、先导实验、仿真或实际执行结果证明方案在一定范围内可行。
12. **参考论文（References）：** 给出真实论文列表，优先使用近五年高质量期刊/会议和领域经典文献。

## Argument and Tone

- 使用中文学术写作风格：严谨、具体、可验证，避免宣传稿、商业计划书或空泛愿景。
- 每个核心判断都应能追溯到数据、文献、公式、实验或明确假设。
- 将“科学假设”写成可被证伪的命题，例如“在 X 条件下，Y 方法相较 Z 基线可提升 A 指标”。
- 区分“已有证据”“待验证假设”“预期结果”和“潜在应用”，不要混写。
- 不夸大社会价值；价值陈述必须与可验证问题和技术路线相连。

## Evidence Planning

- 至少规划一个强基线和一个简单基线；机器学习任务应包含传统统计/规则方法与主流模型对照。
- 指标必须与研究目标匹配：分类用 Accuracy/F1/AUC，回归用 MAE/RMSE/R2，排序用 NDCG/MAP，因果问题用 ATE/CATE 或稳健性检验。
- 数据集部分必须说明合规性、缺失值、偏差来源、样本代表性和隐私风险。
- Source 与 Target 要形成闭环：历史数据用于提出假设，拟采集数据用于验证假设。
- Results 可以是阶段性，但必须给出公式、表格、数值范围或实验日志，而不是只写“预计可行”。

## Outline Guidance

- 必须保留 12 个标准化字段，字段名称不得随意合并或省略。
- 优先把逻辑链条写清楚：Problem Statement -> Source -> Rationale -> Target -> Methods -> Experiments -> Results。
- Technical Details 不只列技术名词，还要说明每项技术服务于哪个假设验证环节。
- References 应在生成前后进行真实性检查；无法核验的文献应移出最终稿。
- 如果证据不足，应明确写为“拟验证”或“预期”，不要伪造成已经完成的实验结果。
